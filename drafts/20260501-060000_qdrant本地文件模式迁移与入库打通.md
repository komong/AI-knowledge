# 问题

前一轮 [embedding 依赖链路修复](./_committed/20260501-050000_embedding依赖链路修复全记录.md) 跑通了 Windows 端 BGE-M3 加载，但真正执行 `scripts.commit` 入库时，在 Qdrant 连接这一层**长时间无法稳定连通**：

- WSL 内 `curl localhost:6333` 200 ok
- Windows 端 `urllib.request` 时好时坏（首次通、几秒后 10061 拒）
- Windows 端 `httpx` / `qdrant-client` **始终** 10061 拒（即使 `trust_env=False`）
- Windows 端 `socket.connect()` 有时通有时拒，状态和 httpx 不一致
- `wsl --shutdown` 后 WSL IP 不变（`172.31.220.236`），但转发恢复失败

最终判定：WSL2 NAT 模式下的 localhost 转发机制对长连接客户端（httpx/grpc）不稳，且容器刚起来那几秒网桥还未就绪。

# 场景

- Windows 11 22H2 + WSL2 Ubuntu（非 mirrored 网络）
- Qdrant 跑在 WSL Docker，Python `.venv` 跑在 Windows
- 已完成：BGE-M3 模型缓存、FlagEmbedding 可加载、依赖版本已锁
- 待完成：3 条草稿入库、MCP Server 自测、Git 提交

# 解决方案

## 1. 根治：Qdrant 改用本地文件模式（推荐）

`qdrant-client` 原生支持无服务端的本地文件模式：

```python
from qdrant_client import QdrantClient
client = QdrantClient(path="./data/qdrant_local")  # 同 API，零网络
```

**改造 `src/store.py`**，按优先级探测连接方式：

```python
url = url or os.getenv("QDRANT_URL")
if url:
    self.client = QdrantClient(url=url)
else:
    path = os.getenv("QDRANT_PATH") or str(
        Path(__file__).resolve().parent.parent / "data" / "qdrant_local"
    )
    self.client = QdrantClient(path=path)
```

**代价与约束**：
- payload index 在本地模式下不生效（warning 可忽略，filter 走暴力扫）
- **同一 `path` 不能被多进程同时打开** → `commit` 脚本与 `MCP Server` 不可并行运行
- 跨机同步靠 `data/qdrant_local/` 目录（已加入 `.gitignore`）

## 2. commit.py 的二级标题误切分 bug

原 `_extract_section` 用 `stripped.startswith("#")` 切分，`##` 也触发 break，导致 040/050 草稿的 `# 解决方案` 下紧跟 `## 方案 x` 立刻 break，captured 为空 → `缺少必要字段`。

修复：仅一级 `# ` 作为切分点：

```python
is_h1 = stripped.startswith("# ")  # 注意空格，严格一级
if is_h1:
    if capturing: break
    if header in stripped: capturing = True
    continue
if capturing: captured.append(line)
```

## 3. FlagEmbedding 内部 snapshot_download 的 403 绕过

即便外层预下载加了 `ignore_patterns=['*.DS_Store','imgs/*']`，`BGEM3FlagModel.__init__` 内部仍会自己调一次 `snapshot_download` 并**触发 hf-mirror 对 `imgs/.DS_Store` 的 403**。

**规避**：运行前设 `HF_HUB_OFFLINE=1`，强制只用本地缓存：

```powershell
$env:HF_HUB_OFFLINE="1"
.\.venv\Scripts\python.exe -m scripts.commit
```

# 已踩过的坑（避免重踩）

- Windows urllib 能连 WSL IP、httpx 同参数就拒 → 不是代理/防火墙而是 httpx 连接池与 WSL2 网桥时序冲突
- `wsl --shutdown` 后 WSL IP 不一定变，但 Windows 侧 portproxy 需要重建
- qdrant 本地模式多进程打开会报 `Storage folder ... is already accessed by another instance`
- 本地模式下 `create_payload_index` 只是 warning，不是 error，但过滤效率下降
- `HF_HUB_OFFLINE=1` 必须在 `import FlagEmbedding` **之前** 设置

# 待解决问题

## 一、未根治的混合环境网络问题

### 1. WSL2 ↔ Windows localhost 转发对 httpx/grpc 不稳
- **现象**：WSL 内 `curl localhost:6333` 200，Windows `urllib` 间歇通，`httpx`/`qdrant-client` 始终被拒 `WinError 10061`
- **根因猜测**：WSL2 NAT 模式的 vEthernet 网桥时序不稳 + httpx 长连接池需要的 TCP 行为与网桥不兼容
- **本轮规避**：改为 Qdrant 本地文件模式
- **彻底解决选项**（未验证）：
  - [ ] 切换 WSL2 **mirrored 网络模式**（`.wslconfig` 加 `networkingMode=mirrored`，需 Win11 22H2+）
  - [ ] 管理员设 `netsh interface portproxy add v4tov4 listenport=6333 connectaddress=<WSL_IP> connectport=6333`
  - [ ] 放弃混合架构，所有服务进 WSL

### 2. WSL IP 每次重启可能变化
- **现象**：`172.31.220.236` 本次没变但跳次不保证；portproxy 需重建
- **解决**：mirrored 模式或写 PowerShell 脚本开机自动重建 portproxy

## 二、Qdrant 本地文件模式的限制

### 3. commit 和 MCP Server 不能并行
- **现象**：同一 `path` 被多进程打开 → `Storage folder ... is already accessed by another instance`
- **当前规避**：读写互斥，跑 `commit` 时关 `server`
- **彻底解决**：一写多读需要回滚到服务端 Qdrant（见一.1）

### 4. payload index 在本地模式下不生效
- **影响**：languages/tags 过滤走暴力扫，体量上千后会明显变慢
- **实测门槛**：未测，估计5k 条之内不明显。到问题阈值时述1 切回服务端
