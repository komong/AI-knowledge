# 草稿审核指南

> 写给非技术背景的用户：如何在不懂技术细节的情况下，高效审核 agent 起草的知识库草稿。

---

## 你的角色：编辑，不是技术评审

| 你在做什么 | 你不是在做什么 |
|-----------|--------------|
| 判断这条经验值不值得存 | 验证技术方案对不对 |
| 检查有没有隐私泄露 | 逐行验算代码逻辑 |
| 确认分类标签合理 | 判断有没有更好的解法 |
| 发现重复/过时内容 | 做性能基准测试 |

你只需把好「门」，好东西放进去，垃圾挡在外面。

---

## 审核三步走（每条 30 秒）

### 第一步：看一眼「一句话总结」

每篇草稿现在都有 `# 一句话总结` section，agent 会用通俗语言解释：

> "发生了什么、为什么重要、怎么解决的"

读完这句你就知道这条大概在说什么了。看不懂？说明 agent 没写好，直接让它改（见下文「怎么让 agent 调整」）。

### 第二步：判断是否值得存

对着这四条问自己：

| # | 问题 | 答「是」→ 存 |
|---|------|-------------|
| 1 | 以后还会再遇到吗？搜到这条有用吗？ | ✅ |
| 2 | 这个解法是非显然的（踩坑才找到的）吗？ | ✅ |
| 3 | tags/languages 标签合理吗？ | ✅ |
| 4 | 正文里有密码、密钥、真实人名吗？ | ❌ 答「是」→ 不存 |

不用关心技术细节是否正确——agent 写给 agent 读的，你只管「复用价值」。

### 第三步：决定

- **值得存** → 终端跑 `python3 -m scripts.commit` 入库
- **不存** → 删掉草稿文件，或移到别处
- **需要改** → 见下文

---

## 怎么让 agent 调整草稿

如果你觉得草稿有问题（太技术看不懂、写太简略、标签不对），**直接告诉当前对话的 agent 就行**。例如：

- "这条一句话总结我看不懂，用更通俗的话重写"
- "solution 写详细点，加上具体的命令"
- "tags 加一个 deployment"

agent 会调用 `edit_draft` 工具直接修改草稿，不需要重写一份。

## 日常命令

```bash
# 看看有多少待审核的草稿
python3 -m src.server   # 启动后 agent 可调 list_drafts

# 预览即将入库的内容（不真入库）
python3 -m scripts.commit --dry-run

# 正式入库（向量化 + 存 Qdrant）
python3 -m scripts.commit

# 检索验证
python3 -c "
from src.store import Store
import json
results = Store().search('你的查询关键词', limit=3)
print(json.dumps(results, ensure_ascii=False, indent=2))
"
```

## 常见场景

**Q: 两条草稿讲的是一件事怎么办？**
A: 删掉旧的那条，留质量好的。入库前看一眼草稿池避免重复。

**Q: agent 写了一堆技术术语看不懂？**
A: 让它改「一句话总结」。总结是给你看的，正文是给 agent 看的，各取所需。

**Q: 不确定该不该存？**
A: 默认存。知识库宁多勿漏，以后用不上也不会碍事。但明显的琐事（修 typo、标准语法）坚决不存。
