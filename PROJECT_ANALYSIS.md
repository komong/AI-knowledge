# Knowledge MCP 项目分析报告

## 项目概述

**Knowledge MCP** 是一个开发经验复利知识库，通过 MCP 协议接入 Claude Code、Cursor、Claude Desktop 等 AI 工具，实现跨 IDE 共享的本地知识库。

核心理念：**复利飞轮** —— 每次解决问题 → agent 起草记忆 → 审核入库 → 下次自动检索复用。

---

## 优点

### 1. 核心理念清晰

"复利飞轮"的概念很实用 —— 每次 debug 都是在积累资产，下次遇到类似问题不用重蹈覆辙。本质上是在构建个人知识资产。

### 2. 本地优先，不依赖云服务

- **BGE-M3** 离线 embedding，不依赖 OpenAI 等外部 API
- **Qdrant** 本地运行，数据完全自主可控
- **drafts** 用 Git 同步，换机器也能重建向量索引

### 3. 设计细节到位

| 设计点 | 说明 |
|--------|------|
| 草稿用 Markdown + frontmatter | 人类可读、git 可追踪、IDE 可编辑 |
| problem 加权 embedding | 查询用问题搜，加权能提升检索准确率 |
| commit 命令不暴露给 agent | 审核闸是物理隔离，不是靠"信任" |

### 4. AGENT_GUIDE.md 是关键

光有工具没用，这个 prompt 引导决定了 agent 什么时候该查、什么时候该存，是飞轮的引擎。

---

## 可改进的地方

### 1. 检索粒度问题

**现状**：目前是整篇文档检索

**问题**：如果一个大文档里只有一小段相关，召回的噪声会比较大

**建议**：
- 按段落/ chunk 拆分文档
- 或者在 frontmatter 中增加 `summary` 字段作为检索目标

### 2. 缺少使用频率追踪

**现状**：没有记录条目的使用次数或最近使用时间

**问题**：长期不用的条目会污染知识库，难以清理

**建议**：在 payload 中增加字段：
```yaml
last_used_at: 2025-05-01  # 上次使用时间
use_count: 5              # 使用次数
```

在 `search_memory` 命中后自动更新，长期没用的可以归档或提示清理。

### 3. 没有去重/合并机制

**现状**：草稿审核后直接入库

**问题**：相似问题可能存了多份，重复内容影响检索质量

**建议**：增加 `scripts/dedupe.py`，对相似度 > 0.9 的旧条目提示合并。

### 4. 跨机器同步成本

**现状**：drafts 用 Git 同步，reindex 是全量的

**问题**：如果草稿上千篇，换机器重建会比较慢

**建议**：
- 考虑增量索引，只处理新增/修改的草稿
- 或者提供 `drafts` 目录的云同步（如 iCloud、Dropbox）

### 5. 缺少批量审核功能

**现状**：需要逐个 review 草稿

**建议**：
- 增加 `commit --dry-run` 查看即将入库的内容
- 增加 AI 辅助批量 review 的脚本

---

## 总结

**评分：7/10**

作为一个"个人开发经验沉淀"的工具，定位清晰、实现简洁、本地优先，是很实用的项目。

核心价值不在于技术多复杂，而在于**养成了一个持续积累的习惯**。技术只是降低了"记"这件事的摩擦，真正难的是坚持 review 和入库。

---

## 优先级建议

| 优先级 | 改进项 |
|--------|--------|
| P1 | 使用频率追踪 |
| P1 | 批量审核功能 |
| P2 | 去重/合并机制 |
| P2 | 增量索引 |
| P3 | 检索粒度优化（chunk 拆分） |
